# 이니셔티브 업데이트 작성 가이드

## 핵심 원칙

이니셔티브 업데이트는 **프로젝트 업데이트의 요약이 아니라**, 이니셔티브 전체 관점에서 "무엇이 달라졌는지"를 전달하는 것

### DO
- 이니셔티브 목표 달성 관점에서 주요 성과 선별
- 프로젝트별 1줄 요약 (핵심 성과만)
- 팀 전체가 이해할 수 있는 비즈니스 언어 사용
- 프로젝트 간 연관성/시너지 언급
- 주요 리스크/이슈의 영향도 설명

### DON'T
- 프로젝트 업데이트 복붙
- 이슈 번호 나열
- 기술 용어만 나열
- 모든 세부 작업 열거
- 개별 프로젝트 관점 서술

---

## 하이라이트 선별 기준

### 성과 선별 기준
1. **마일스톤 달성**: 주요 기능 완료, 릴리즈, 런칭
2. **전환점**: 기술 선택, 아키텍처 결정, 방향 전환
3. **임팩트**: 사용자/비즈니스에 즉각 영향 미치는 변화
4. **블로커 해소**: 여러 프로젝트에 영향 미치던 이슈 해결

### 리스크/이슈 선별 기준
1. **일정 영향**: 전체 이니셔티브 일정에 영향
2. **범위 변경**: 목표나 스코프 조정 필요
3. **리소스 이슈**: 인력/예산 문제
4. **외부 의존성**: 다른 팀/시스템 영향

## Health 자동 종합 로직

하위 프로젝트들의 health를 기반으로 이니셔티브 health 자동 결정:

| 프로젝트 상태 | 이니셔티브 Health |
|--------------|------------------|
| 모든 프로젝트 onTrack | 🟢 onTrack |
| 1개라도 atRisk | 🟡 atRisk |
| 1개라도 offTrack | 🔴 offTrack |

**수동 조정 가능한 경우:**
- 프로젝트 health가 오래 전 업데이트인 경우
- 최근 상황 변화가 반영 안 된 경우
- 리드가 판단한 실제 상황과 다른 경우

## 좋은 예 vs 나쁜 예

### 핵심 성과 작성

**❌ 나쁜 예:**
```
- [프로젝트A] CORE-123, CORE-124, CORE-125 완료
- [프로젝트B] API 엔드포인트 구현, DB 스키마 변경
```

**✅ 좋은 예:**
```
- [프로젝트A] 결제 시스템 v2 베타 오픈 → 내부 테스트 시작
- [프로젝트B] 외부 파트너 API 연동 완료 → 다음 주 파일럿 예정
```

### 리스크/이슈 작성

**❌ 나쁜 예:**
```
- 프로젝트C 지연
- 인력 부족
```

**✅ 좋은 예:**
```
- [프로젝트C] 외부 API 인증 이슈로 연동 1주 지연 → 전체 일정에는 영향 없음
- 시니어 개발자 휴가로 2주간 리뷰 병목 예상 → 코드 리뷰 프로세스 임시 조정
```

## 프로젝트-이니셔티브 연결 표현

프로젝트 성과를 이니셔티브 목표와 연결:

```
이니셔티브 목표: "Q1 결제 시스템 현대화"

프로젝트 성과:
- [결제 API] 신규 결제 수단 3종 추가 → 목표 대비 60% 달성
- [정산 시스템] 실시간 정산 기능 완료 → 파트너 만족도 개선 기대
```
